# 20강
1. 단계. Persistent Volumes 생성
- 참조 URL : https://kubernetes.io/docs/concepts/storage/persistent-volumes/#persistent-volumes
- Concepts > Storage > Persistent Volumes
 
- 네트워크 파일 시스템(NFS)을 사용하여 /tmp 디렉토리에 접근할 수 있는 PV를 정의
``````
  apiVersion: v1  
  kind: PersistentVolume  # 생성할 객체의 형식 PersistentVolume
  metadata:
    name: pv0003          # 사용 이름
  spec:
    capacity:
      storage: 5Gi        # 저장 용량 5Gi( GiB (Gibibyte): 1 GiB는 1,073,741,824 바이트, GB (Gigabyte): 1 GB는 1,000,000,000 바이트)
    volumeMode: Filesystem
    accessModes:
      - ReadWriteOnce     # 한 번에 하나의 Pod만 이 PV에 쓰기 및 읽기 작업 (허용할 액세스 모드)
    persistentVolumeReclaimPolicy: Recycle
    storageClassName: slow
    mountOptions:         # NFS 마운트가 하드 마운트로 설정, 연결이 실패할 경우 NFS 클라이언트는 지속적으로 재시도
      - hard
      - nfsvers=4.1
    nfs:                  # 실제 저장할 매체가 어디에 있는지 지정(네트워크 , 호스팅된 파일경로 , efs 또는 evs 볼륨과 같은 클라우드
스토리지 등)
      path: /tmp          # 접근경로
      server: 172.17.0.2  # 접근 서버의 IP
``````
2. 단계. PV Claims 생성
- 참조 URL : https://kubernetes.io/docs/concepts/storage/persistent-volumes/#persistentvolumeclaims
- PersistentVolume(PV)을 요청하는 클레임을 정의
```
  apiVersion: v1
  kind: PersistentVolumeClaim # PersistentVolumeClaim(PVC)
  metadata:
    name: myclaim             # PVC의 이름
  spec:
    accessModes:
      - ReadWriteOnce
    volumeMode: Filesystem
    resources:
      requests:
        storage: 8Gi
    storageClassName: slow
    selector:
      matchLabels:
        release: "stable"     # 이 PVC는 release라는 레이블이 "stable"로 설정된 PV와만 바인딩
      matchExpressions:
        - {key: environment, operator: In, values: [dev]}
```
3. 단계. 해당 볼륨을 실제적으로 사용하는 방법
- 참조 URL : https://kubernetes.io/docs/concepts/storage/persistent-volumes/#claims-as-volumes
- Pod는 클레임을 볼륨으로 사용하여 저장소에 액세스합니다. 클레임은 클레임을 사용하는 Pod와 동일한 네임스페이스에 있어야 합니다. 클러스터는 Pod의 네임스페이스에서 클레임을 찾아 클레임을 지원하는 PersistentVolume을 가져오는 데 사용합니다. 그런 다음 볼륨이 호스트와 Pod에 마운트됩니다.
```
  apiVersion: v1
  kind: Pod
  metadata:
    name: mypod
  spec:
    containers:
      - name: myfrontend
        image: nginx
        volumeMounts:
        - mountPath: "/var/www/html"
          name: mypd
    volumes:
      - name: mypd
        persistentVolumeClaim:
          claimName: myclaim
```

#### 고려사항
- 참조 URL : https://kubernetes.io/docs/concepts/storage/persistent-volumes/#mount-options
- Mount Options : Kubernetes 관리자는 영구 볼륨이 노드에 마운트될 때 추가 마운트 옵션을 지정할 수 있다. 
- 필요한 물리적인 드라이버를 사용하여 연결한다

# 21강
#### 개요
1. Service 란?
- Service는 k8s에서 여러 개의 파드(애플리케이션 인스턴스)를 하나의 네트워크 서비스로 묶어 외부나 클러스터 내부에서 접근 할 수 있게 해주는 객체
- 예) 5개의 pod가 있다면, Service가 로드밸런서 역할을 하여 하나의 IP나 도메인으로 모든 파드에 요청을 분산시켜 준다
2. Service 의 요청 분산 방법
- Service는 라벨(Label)을 기준으로 파드를 선택
- 라벨과 셀렉터를 이용한 서비스 연결
  - 파드에 ,app=myapp, version=1과 같은 라벨이 붙어 있다
  - Service는 selector를 통해서 어떤 라벨을 가진 파드에 요청을 보낼지 지정 app=myall 라벨을 가진 5개의 pod가 있다면 Service는 이중에 하나를 골라 요청을 전달 한다
3. 서비스의 타입 종류 특징
- https://kubernetes.io/docs/concepts/services-networking/service/#defining-a-service
```
  apiVersion: v1
  kind: Service
  metadata:
    name: my-service
  spec:
    selector:
      app.kubernetes.io/name: MyApp
    ports:
      - protocol: TCP
        port: 80
        targetPort: 9376
```

# 22강
#### 개요
1. Ingress란?
 - Ingress를 통한 여러 개의 마이크로서비스를 경로(Path)나 호스트(host) 기반으로 라우팅 하는 방법
 - 요청 URL의 경로나 호스트 이름에 따라서 서비스로 연결해 준다
2. 호스팅 기반의 라우팅
 - api.***product***.mycompany.com - > 제품서비스
 - api.***promotion***.mycompany.com - > 제품 프로모션
3. 경로 기반의 라우팅
 - mycompany.com/***product*** -> 제품서비스
 - mvcompany.com/***promotion*** -> 제품프로모션

4. Ingress 설정 예시
 - Path 요청: /product
 - pathType: 경로매칭 방법(prefix사용)
 - Backend service: 요청을 전달할 k8s서비스 이름과 포트
5. NGINX Ingress Controller
 - K8s는 기본에는 Ingress 기능이 없기 때문에 Ingress를 실제로 작동시키려면 Ingress Controller가 필요하다
 - NGINX Ingress Controller를 사용하여 Ingress 리소스 설정을 통해서 요청을 서비스로 전달
 - 실습 API와 UI Ingress 리소스 설정
   - API Ingress
      - /bookmarker-api로 시작하는 요청을 bookmarker-api-svc로 전달
   -  Ul Ingress
      - / 요청을 bookmaker-ui-svc 서비스로 전달
   - 즉, bookmaker-api로 시작하는 요청을 백엔드 API로 그 외의 요청은 UI로 요청 처리
